## 这次关键问题与根因映射
1. 现象：`REMOTE HOST IDENTIFICATION HAS CHANGED`
- 根因：本地 `known_hosts` 残留旧指纹。
- 标准处理：`ssh-keygen -R <IP>`。

2. 现象：`server add` 显示完成但免密失败。
- 根因：服务端 `sshd` 配置 `pubkeyauthentication no`。
- 标准处理：启用 `PubkeyAuthentication yes`，并重载 `sshd`。

3. 现象：命令长时间卡住。
- 根因：SSH在等待交互认证路径（并非业务命令本身慢）。
- 标准处理：先用 `BatchMode=yes` 无交互验证认证路径。

## 可复用执行模板（建议直接复制）
```bash
# A. 首次接入
python sshctrl.py server add <IP> <用户> <密码> <别名>

# B. 若免密失败，自动修复
python sshctrl.py server repair-pubkey <别名> <密码>

# C. 强制验收（必须通过）
ssh -o BatchMode=yes <别名> "whoami && hostname"

# D. 运维状态采集
ssh <别名> "date '+%F %T %Z' && uptime && free -h && df -h /"
```

## 经验抽象（适用于所有服务器）
1. “端口可达”不等于“认证可用”。
2. `server add` 的成功提示不应视为最终成功，必须加无交互验收。
3. 排查顺序应固定：
   - 指纹冲突 -> 认证方式 -> 服务端策略 -> 业务命令。
4. 自动化要覆盖“异常分支”，而不仅是 happy path。

## 交付验收口径（推荐）
同时满足以下条件才算完成：
1. `ssh -o BatchMode=yes <别名> "echo ok"` 成功。
2. `ssh <别名> "whoami && hostname"` 正常返回。
3. `sshd -T` 中 `pubkeyauthentication yes`。
4. 关键资源状态可采集（CPU/内存/磁盘/负载）。
